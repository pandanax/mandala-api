# Планы и квоты

## Требования

- Лимиты **настраиваются** без изменения кода: число текстовых ответов, число картинок (в т.ч. **0**).
- Разные планы (free / paid / промо) задаются в БД или конфиге, который заливается миграцией.

## Таблица `plans`

- `id`
- `name` (`free`, `premium`, …)
- `description` (опционально)
- Для привязки к оплате: `billing_provider`, `external_product_id` (Stars / Stripe price id и т.д.)

## Таблица `plan_limits`

- `plan_id` → `plans`
- `resource` — перечисление: например `text_reply`, `image_generation`
- `limit_per_period` — целое число (**0** означает запрет операции в этом плане)
- `period` — `week` | `month` | `year` | `lifetime` (наследие для free)

Семантика: за календарный или биллинговый период пользователь не может превысить лимит по данному ресурсу.

## Учёт usage

- Перед операцией: в транзакции прочитать лимит плана и текущий счётчик за период.
- Если `current >= limit` — отказ с понятным сообщением и предложением апгрейда (Stars).
- После успешной генерации — атомарный инкремент `usage_counters`.

Рекомендация: **`SELECT … FOR UPDATE`** на строке счётчика или единая транзакция «проверка + инкремент» для защиты от гонок.

Реализация в репозитории: атомарный **`UPDATE … WHERE count < limit`** (`UsageRepository.try_increment`); сервис **`mandala.services.quota.QuotaService.consume`** использует его для инкремента.

| План | text_reply / месяц | image_generation / месяц |
|------|---------------------|---------------------------|
| free | 20 | 1 |
| premium | 200 | 10 |
| trial | 50 | 0 |

Числа задаются только в данных, не в коде.

## Смена плана

При апгрейде уточнить продуктово:

- обнулять ли счётчики текущего периода;
- или пересчитывать только с следующего периода.

Зафиксировать правило в коде одной функцией `apply_plan_change(user_id, new_plan_id, reason)`.
