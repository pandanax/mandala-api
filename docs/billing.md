# Биллинг

## Принцип

Бизнес-логика не зависит от конкретного способа оплаты. Все провайдеры реализуют общий контракт; активация тарифа происходит в **одном месте** после подтверждённого платежа.

## Интерфейс BillingProvider (логический)

Методы (названия условные):

- `create_payment_offer(user_id, plan_id, context)` — что нужно каналу для показа оплаты (например данные для Telegram invoice).
- `handle_pre_checkout(payload)` — валидация перед списанием (Telegram).
- `handle_successful_payment(payload)` — после успеха: идемпотентная запись транзакции + обновление плана.
- При необходимости: `refund`, `sync_subscription` для Stripe и т.д.

## Telegram Stars (первая реализация)

- Обработка **`pre_checkout_query`** — ответ без блокировки пользователя при корректном тарифе.
- Обработка **`successful_payment`** — извлечение суммы, валюты, идентификаторов; вызов доменного сервиса «активировать план».
- Связь продуктов Stars с **`plans`** и `external_product_id` в БД (настраиваемо).

Требования:

- **Идемпотентность**: повторный webhook с тем же платежом не продлевает подписку второй раз.
- Логи без лишних PII; сырые payload — ограниченный доступ.

## Другие провайдеры (задел)

| Провайдер | Особенности |
|-----------|-------------|
| Stripe | Подписки, webhooks, customer portal |
| ЮKassa | РФ, разные сценарии оплаты |

Общая таблица транзакций (`payment_transactions`) с полем `provider` и уникальностью по `(provider, external_id)`.

## Связь с планами

После успешной оплаты:

1. Запись в `payment_transactions` со статусом `completed`.
2. Обновление `users.current_plan_id` и границ периода подписки (или отдельная таблица `subscriptions`).
3. Опционально: сброс или перенос **usage** на новый период — по правилам продукта.
