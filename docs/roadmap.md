# План развития (после MVP)

Краткий ориентир: что логично делать дальше, когда текущий **MVP** (Telegram + HTTP, PostgreSQL, LLM, квоты, Stars) уже в проде и стабилен. Детали реализации — отдельными задачами и PR.

## Надёжность и команда

- **Terraform state** в **Object Storage** (YC), блокировки state — чтобы несколько человек безопасно делали `plan` / `apply`.
- **Секреты** в **Lockbox** (или аналог) и выдача на ВМ при старте, а не только файл на диске.
- **Мониторинг и алерты** (здоровье API, БД, latency LLM, ошибки webhook).

## Продукт и каналы

- Расширение **Web**-канала (авторизация, ключи к вертикали вместо заголовков MVP).
- Дополнительные вертикали и сценарии без ломки ядра (**`vertical_id`**, конфиги, KB).

## Инфраструктура

- Отдельная выкладка образа (registry, теги, откат).
- При росте нагрузки — вынос БД/приложения или managed-сервисы по метрикам.
- **RAG** в проде: Qdrant или managed-вектор, индексация по расписанию.

## Документация

- Поддерживать **актуальность** [data-model.md](data-model.md) относительно кода.
- Публичный **OpenAPI** / страница «для интеграторов» при появлении внешних клиентов Web API.
