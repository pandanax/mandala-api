# Агент: оркестрация, модели, RAG, память

## Роль

Агент реализует **сценарий конкретной вертикали** (`vertical_id`): сбор полей профиля (набор шагов и валидации зависят от агента), опора на **базу знаний этой вертикали** (не только общий промпт), генерация текстовых ответов и при разрешении тарифа — **изображений**. Платформа переиспользует один движок графа и роутинга; содержимое сценария и промптов задаётся конфигом **вертикали** (см. [data-model.md](data-model.md), `agent_card` / `scenario_state` в JSONB).

Пример: «мандала» — одна из вертикалей; астролог / психолог — другие, с другими шагами и KB.

## Контекст выполнения

Каждый запуск графа и вызов LLM опирается на:

- **`vertical_id`** — выбор конфига (промпты, список шагов, пути к KB, дефолтные модели);
- **`user_id`** — загрузка профиля и истории в разрезе этой вертикали;
- **`InboundEvent`** — уже с заполненным `vertical_id` (см. [channels.md](channels.md)).

В логах и метриках желательно всегда указывать **`vertical_id`** вместе с `user_id` (без лишнего PII в тексте сообщений).

## Оркестрация

Рекомендуется **граф состояний** (например **LangGraph**):

- Узлы и порядок шагов **частично или полностью задаются конфигом вертикали** (приветствие, сбор полей, RAG-ответ, image API, финал) — не обязательно одна жёсткая цепочка «как для мандалы».
- Рёбра: условия по заполненности **`scenario_state` / `agent_card`** в профиле клиента и по намерению (продолжить анкету / вопрос / картинка).

Альтернатива: собственный минимальный граф на **asyncio** + **Pydantic** для состояния — если нужно меньше зависимостей.

## Роутинг моделей

Отдельный слой **Router** (или узел графа):

- По типу задачи: **диалог** → текстовая модель, **изображение** → image API.
- Перед вызовом **image** — проверка квоты `image_generation` и что `limit_per_period` > 0 в **плане пользователя этой вертикали**.

Реализация: интерфейсы `TextCompletionClient`, `ImageGenerationClient` с конкретными бэкендами; дефолтные модели могут переопределяться конфигом **`vertical_id`**.

## База знаний (RAG)

- Документы и чанки **изолированы по `vertical_id`**: индексация и запрос к векторному хранилищу всегда с фильтром по вертикали (чтобы KB астролога не попадал в ответы психолога).
- Пайплайн: чанкинг → эмбеддинги → векторный поиск по запросу пользователя (и при необходимости по терминам из **разрешённых полей профиля**).
- В промпт попадают **top-k** чанков только своей вертикали + краткий контекст из профиля (то, что сценарий уже разрешил выдавать в модель).

Хранилище эмбеддингов: **отдельный слой** от OLTP (pgvector, Qdrant и т.д.) — см. [architecture.md](architecture.md).

## Память диалога

- Краткосрочно: последние N сообщений в контексте модели.
- Долгосрочно: таблица **`messages`** с полем **`vertical_id`** (или эквивалентная изоляция) + периодический **summary** для укладывания в лимит токенов.

Профиль пользователя в БД (**`agent_card`**, факты о человеке) — не путать с RAG: это **данные клиента и сценария**, не учебные материалы вертикали.

## Долгие задачи (картинки)

Генерация изображения может занимать заметное время:

- Вариант A: очередь (worker), пользователю — сообщение «генерирую», затем **OutboundMessage** с фото.
- Вариант B: короткий polling в Web; для Telegram — push по готовности.

Сохранение результата (метаданные, ссылки, `file_id`) — в **`messages` / `generated_artifacts`** с привязкой к **`user_id`** и **`vertical_id`**.

## Наблюдаемость

Логировать: **`vertical_id`**, переходы графа, выбранный провайдер, latency, ошибки API (без сырых PII в тексте переписки). Опционально трейсы OpenTelemetry для продакшена.
