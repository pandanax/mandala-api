# Агент: оркестрация, модели, RAG, память

## Роль

Агент реализует **сценарий конкретной вертикали** (`vertical_id`): сбор полей профиля (набор шагов и валидации зависят от агента), опора на **базу знаний этой вертикали** (не только общий промпт), генерация текстовых ответов и при разрешении тарифа — **изображений**. Платформа переиспользует один движок графа и роутинга; содержимое сценария и промптов задаётся конфигом **вертикали** (см. [data-model.md](data-model.md), `agent_card` / `scenario_state` в JSONB).

Пример: «мандала» — одна из вертикалей; астролог / психолог — другие, с другими шагами и KB.

## Контекст выполнения

Каждый запуск графа и вызов LLM опирается на:

- **`vertical_id`** — выбор конфига (промпты, список шагов, пути к KB, дефолтные модели);
- **`user_id`** — загрузка профиля и истории в разрезе этой вертикали;
- **`InboundEvent`** — уже с заполненным `vertical_id` (см. [channels.md](channels.md)).

В логах и метриках желательно всегда указывать **`vertical_id`** вместе с `user_id` (без лишнего PII в тексте сообщений).

## Оркестрация

Минимальный текстовый контур (**тикет 12**): после идентификации пользователя — при **незавершённой анкете** вертикали (тикет 13) вопросы и валидация по конфигу шагов, запись ответов в **`agent_card`** и прогресса в **`scenario_state`**; после **`intake_complete`** — сохранение реплики в **`messages`**, квота **`text_reply`**, вызов **`TextCompletionClient`**, ответ ассистента и **`consume`** (**`mandala.services.text_reply`**). Роутер **текст / картинка** (**тикет 14**): **`post_intake_intent`** → при запросе изображения квота **`image_generation`** и **`ImageGenerationClient`**; **тикет 15** — HTTP к **`/images/generations`** (или заглушка), **`generated_artifacts`**, **`sendPhoto`** при наличии URL. Асинхронная очередь под долгие задачи — см. **architecture.md**.

Рекомендуется **граф состояний** (например **LangGraph**):

- Узлы и порядок шагов **частично или полностью задаются конфигом вертикали** (приветствие, сбор полей, RAG-ответ, image API, финал) — не обязательно одна жёсткая цепочка «как для мандалы».
- Рёбра: условия по заполненности **`scenario_state` / `agent_card`** в профиле клиента и по намерению (продолжить анкету / вопрос / картинка).

Альтернатива: собственный минимальный граф на **asyncio** + **Pydantic** для состояния — если нужно меньше зависимостей.

## Роутинг моделей

Отдельный слой **Router** (или узел графа):

- По типу задачи: **диалог** → текстовая модель, **изображение** → image API (**тикеты 14–15**: **`ImageGenerationClient`**, запись **`messages`** / **`generated_artifacts`**).
- Перед вызовом **image** — проверка квоты `image_generation` и что `limit_per_period` > 0 в **плане пользователя этой вертикали** (при **0** клиент генерации **не вызывается**).

Реализация: интерфейсы **`TextCompletionClient`**, **`ImageGenerationClient`** с конкретными бэкендами; дефолтные модели могут переопределяться конфигом **`vertical_id`**. В коде: пакет **`mandala.llm`** (`OpenAICompatibleTextClient`, **`OpenAICompatibleImageClient`**, **`StubImageGenerationClient`**, **`LlmConfigProvider`**, **`LlmProviderError`** — см. тикеты 11, 14–15 в `implementation-plan.md`).

## База знаний (RAG)

- Документы и чанки **изолированы по `vertical_id`**: при поиске в Qdrant всегда задаётся фильтр по вертикали (смоук-тест: `tests/test_rag_vertical_isolation.py`).
- Пайплайн: сырьё в репозитории **`mandala/verticals/kb/{vertical_id}/`** (или корень из **`MANDALA_KB_ROOT`**) → чанкинг → эмбеддинги (**`POST {LLM_BASE_URL}/embeddings`**, модель **`LLM_EMBEDDING_MODEL`**) → запись в **Qdrant**; перед вызовом чат-модели — retrieval и вставка фрагментов в **системный** промпт (**`mandala.services.text_reply`**).
- В промпт попадают **top-k** чанков только своей вертикали (переменная **`RAG_TOP_K`**, по умолчанию 5). Учёт лимита контекста RAG — по **символам**: **`RAG_MAX_CONTEXT_CHARS`** (по умолчанию 8000); это **не** подсчёт токенов модели — общий бюджет токенов задаётся провайдером и параметром `max_tokens` вызова (**1024** в **`text_reply`**). История диалога (тикет 17) режется **по числу сообщений** (**`TEXT_REPLY_CONTEXT_MESSAGES`** = 20 в **`mandala.services.text_reply`**), а не по символам; суммарный размер запроса к провайдеру = системный блок (вертикаль + KB + опциональная сводка) + до 20 последних реплик из **`messages`** + лимит генерации **`max_tokens`**.

Индексация: **`python -m mandala.index_kb --vertical <slug>`** при **`MANDALA_RAG_BACKEND=qdrant`** и **`QDRANT_URL`**. Runtime: те же переменные; при **`MANDALA_RAG_BACKEND=none`** (дефолт) ответы идут без RAG, без обязательного Qdrant.

Хранилище эмбеддингов: **отдельный слой** от OLTP — см. [architecture.md](architecture.md).

## Память диалога (тикет 17)

- Краткосрочно: последние **N** строк из **`messages`** (после записи текущего входа пользователя) подмешиваются в **Chat Completions** в хронологическом порядке; **N** зафиксировано константой **`TEXT_REPLY_CONTEXT_MESSAGES`** (= **20**) в **`mandala.services.text_reply`**. В роли **`system`** одним сообщением идут: промпт вертикали → фрагменты KB (если RAG включён) → опционально текст из **`scenario_state["dialog_summary"]`** (чтение в **`handle_inbound`**; запись/обновление сводки — вне scope тикета 17).
- Долгосрочно: таблица **`messages`** с **`vertical_id`** + при необходимости сводка в JSONB профиля; отдельная **нарезка истории по токенам** (если понадобится продуктово) — не введена в тикете 17.

Профиль пользователя в БД (**`agent_card`**, факты о человеке) — не путать с RAG: это **данные клиента и сценария**, не учебные материалы вертикали.

## Долгие задачи (картинки)

Генерация изображения может занимать заметное время. **Тикет 15 (MVP):** синхронный вызов image API в том же процессе, что и webhook/polling; провайдер по умолчанию — заглушка без сети. **Дальше:** очередь (worker), ответ «генерирую» отдельным сообщением до готовности фото — см. **architecture.md**.
