## Слои

```

│  Adapters (Telegram, Web, CLI, …)                           │

│  Выход: OutboundMessage → отправка в канал                 │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  Application / Use cases                                   │
│  Идентификация пользователя, квоты, биллинг-хуки,           │

                            │

│  Agent (оркестрация)                                       │
│  Граф состояний (напр. LangGraph): сбор данных, RAG,       │

└───────────────────────────┬─────────────────────────────────┘
                            │

│  Domain + PostgreSQL (гибрид)                              │

                            │
┌───────────────────────────▼─────────────────────────────────┐
│  Integrations                                              │
│  LLM-клиенты, image API, embeddings,                       │
│  векторное хранилище (RAG) — отдельно от OLTP Postgres      │
└─────────────────────────────────────────────────────────────┘
```

## Хранение данных: гибрид PostgreSQL + JSONB и RAG отдельно

**PostgreSQL** — единый OLTP-источник правды для:

- идентичностей, каналов, планов, лимитов, usage, платежей;
- истории сообщений и **учётных полей** артефактов.

**JSONB** используется там, где у разных **агентов / вертикалей** разная форма данных: «карточка клиента», шаги сценария, структурированные рекомендации, произвольные вложения в метаданные — без жёсткой нормализации под каждый тип агента. В приложении — валидация и при необходимости **версия схемы** внутри JSON.

**Векторный поиск (RAG)** вынесен в **отдельное хранилище** (тикет 16: по умолчанию **Qdrant**, не в таблицах OLTP Postgres). Эмбеддинги и ANN-поиск не смешиваются с учётными таблицами; связь с доменом — через **`vertical_id`** в payload точек и при поиске (обязательный фильтр), плюс `source_path` / `chunk_index` в payload. Альтернативы: pgvector в отдельной БД, managed-сервис в облаке — по этапу деплоя (тикет 23).

## Стек (ориентир)

| Компонент | Технология |
|-----------|------------|
| Runtime | Python 3.11+ |
| HTTP / webhook | **FastAPI** с uvicorn (тикет 10: health, webhook endpoints) |
| Telegram | **httpx** + Bot API (тикет 9: long polling, тикет 10: webhook); при необходимости **aiogram** / **python-telegram-bot** — позже |
| Оркестрация агента | LangGraph (или тонкий свой граф на asyncio + Pydantic) |
| OLTP / основная БД | **PostgreSQL** (реляционные таблицы + **JSONB** для гибких тел документов) |
| Кэш / сессии (опционально) | Redis |
| Векторное хранилище (RAG) | **Отдельный слой** от OLTP: в коде MVP зафиксирован **Qdrant** (клиент `qdrant-client`, см. тикет 16); pgvector / Chroma / managed — возможные альтернативы на следующих этапах |
| Очереди для долгих задач (опционально) | Redis + ARQ / Celery для генерации картинок (альтернатива — синхронный вызов в HTTP, тикет 15 MVP) |

## HTTP-вход (тикеты 10, 21)

Пакет **`mandala.http`**: единая точка **`GET /health`**, **`POST /webhooks/telegram/{vertical_id}`** и **`POST /webhooks/web`**. Telegram-webhook не дублирует маппинг: использует **`mandala.adapters.telegram.telegram_update_to_inbound_event`**, затем **`handle_inbound`**, затем **`deliver_outbound_messages`** и **`TelegramBotApiClient`**. Web-эндпоинт строит **`InboundEvent`** с **`channel=web`** (**`mandala.adapters.web`**) и вызывает тот же **`handle_inbound`**; ответ — JSON со списком **`OutboundMessage`** (без UI и без внутреннего **`user_id`** в теле ответа). См. **README** и **docs/channels.md**.

## Поток: входящее сообщение

1. Адаптер канала преобразует событие в **InboundEvent** и определяет **`vertical_id`**, `channel` + `external_user_id`.
2. Сервис приложения находит или создаёт **внутренний `user_id`** по `channel_links` в границах вертикали.
3. Проверка **квоты** (``mandala.services.quota.QuotaService``: ``can_consume`` / ``consume``) — до вызова дорогой модели (**после** завершения анкеты по тикету 13 для текстового LLM-ответа).
4. Загрузка **профиля** (реляция + JSONB) и контекста сценария; при включённом RAG — **retrieval из Qdrant** (тикет 16) с фильтром по **`vertical_id`** и подмешиванием фрагментов в системный промпт перед LLM; для текста (тикет 17) — последние **N** реплик из **`messages`** в контекст чата (константа **`TEXT_REPLY_CONTEXT_MESSAGES`** в **`text_reply`**).
5. Запуск **графа агента** (анкета по конфигу вертикали → диалог); роутер выбирает модель (**тикет 14**: текст vs изображение по **`intent_router`**, квота **`image_generation`** до вызова **`ImageGenerationClient`**).
6. Сохранение сообщений / артефактов; инкремент **usage** при успешной генерации (текст и картинка: **`consume`** после успешного ответа провайдера, см. **`quotas-and-plans.md`** и **`mandala.services.image_reply`**).
7. Формирование **OutboundMessage**; адаптер отправляет в канал (текст, фото по URL в **`photo`** — Telegram **`sendPhoto`**).

## Генерация изображений (тикет 15, MVP)

По умолчанию в env провайдер **`IMAGE_GENERATION_PROVIDER=stub`**: без HTTP, в Telegram уходит только текст-описание. Реальный вызов: **`openai_compatible`** — **один** синхронный **POST** к **`/images/generations`** (формат OpenAI, в ответе **URL**), затем доставка фото в Telegram и запись в **`messages`** + **`generated_artifacts`**. **Без** отдельной очереди Redis в MVP: обработчик webhook/polling **блокируется** на время запроса к image API (таймаут чтения ~180 с). Варианты с **worker** и промежуточным сообщением «генерирую…» (отдельная доставка до конца генерации) — на следующие итерации, если появятся SLO на время ответа.

## Поток: оплата (Stars)

1. Пользователь инициирует оплату в Telegram → адаптер получает `pre_checkout_query` / `successful_payment` → **`process_telegram_billing_update`** → **`handle_pre_checkout_query`** / **`handle_successful_payment`** (тикет 19).
2. Вызов **`PostgresBillingProvider.activate_plan`**: идемпотентная запись в **`payment_transactions`**, при первом успехе — **`UPDATE users`**; затем **`apply_plan_change`** (сброс **usage** за месяц, **`subscription_period_end`**).
3. Уведомление пользователя через **``sendMessage``** (короткий текст после оплаты).

## Наблюдаемость (тикет 20)

Операционные логи воронки **вход → квота → LLM/картинка → доставка** и ветки оплаты: модуль **`mandala.observability`** (**`op_format`**, **`mask_api_key`**), префикс **`funnel`** в сообщениях; на **INFO** не пишутся текст переписки и секреты (токен бота — **`mask_bot_token`** в адаптере Telegram).

## Расширения без ломки ядра

- Новый агент / вертикаль: конфиг + запись в `agent_verticals` (или аналог), отдельные правила валидации JSONB при необходимости.
- Новый канал: новый адаптер + регистрация в `channel_links`.
- Новый платёжный провайдер: новая реализация `BillingProvider`.
- Новый тип лимита: новая строка в `plan_limits` и ресурс в коде учёта usage.
