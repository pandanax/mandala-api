# Архитектура

## Слои

```
┌─────────────────────────────────────────────────────────────┐
│  Adapters (Telegram, Web, CLI, …)                           │
│  Вход: raw событие канала → InboundEvent                    │
│  Выход: OutboundMessage → отправка в канал                 │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  Application / Use cases                                   │
│  Идентификация пользователя, квоты, биллинг-хуки,           │
│  вызов агента, запись usage                                │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  Agent (оркестрация)                                       │
│  Граф состояний (напр. LangGraph): сбор данных, RAG,       │
│  роутинг моделей (текст / картинка)                       │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  Domain + PostgreSQL (гибрид)                              │
│  Реляционное ядро: users, channel_links, plans, usage,     │
│  payments, messages (шапка + ссылки)                        │
│  JSONB: карточка клиента, scenario_state, артефакты/реком.   │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│  Integrations                                              │
│  LLM-клиенты, image API, embeddings,                       │
│  векторное хранилище (RAG) — отдельно от OLTP Postgres      │
└─────────────────────────────────────────────────────────────┘
```

## Хранение данных: гибрид PostgreSQL + JSONB и RAG отдельно

**PostgreSQL** — единый OLTP-источник правды для:

- идентичностей, каналов, планов, лимитов, usage, платежей;
- истории сообщений и **учётных полей** артефактов.

**JSONB** используется там, где у разных **агентов / вертикалей** разная форма данных: «карточка клиента», шаги сценария, структурированные рекомендации, произвольные вложения в метаданные — без жёсткой нормализации под каждый тип агента. В приложении — валидация и при необходимости **версия схемы** внутри JSON.

**Векторный поиск (RAG)** вынесен в **отдельное хранилище** (Qdrant, pgvector в Postgres, managed-сервис в облаке — выбор по этапу). Эмбеддинги и ANN-поиск не смешиваются с требованием «всё в одной таблице»; связь с доменом — через `vertical_id`, `document_id`, `chunk_id` в метаданных.

## Стек (ориентир)

| Компонент | Технология |
|-----------|------------|
| Runtime | Python 3.11+ |
| HTTP / webhook | **FastAPI** с uvicorn (тикет 10: health, webhook endpoints) |
| Telegram | **httpx** + Bot API (тикет 9: long polling, тикет 10: webhook); при необходимости **aiogram** / **python-telegram-bot** — позже |
| Оркестрация агента | LangGraph (или тонкий свой граф на asyncio + Pydantic) |
| OLTP / основная БД | **PostgreSQL** (реляционные таблицы + **JSONB** для гибких тел документов) |
| Кэш / сессии (опционально) | Redis |
| Векторное хранилище (RAG) | **Отдельный слой**: pgvector / Qdrant / Chroma / managed — по этапу и хостингу |
| Очереди для долгих задач (опционально) | Redis + ARQ / Celery для генерации картинок |

## HTTP-вход (тикет 10)

Пакет **`mandala.http`**: единая точка **`GET /health`** и **`POST /webhooks/telegram/{vertical_id}`**. Webhook не дублирует маппинг Telegram: использует **`mandala.adapters.telegram.telegram_update_to_inbound_event`**, затем **`handle_inbound`**, затем **`deliver_outbound_messages`** и **`TelegramBotApiClient`**. См. **README** и **docs/channels.md**.

## Поток: входящее сообщение

1. Адаптер канала преобразует событие в **InboundEvent** и определяет **`vertical_id`**, `channel` + `external_user_id`.
2. Сервис приложения находит или создаёт **внутренний `user_id`** по `channel_links` в границах вертикали.
3. Проверка **квоты** (``mandala.services.quota.QuotaService``: ``can_consume`` / ``consume``) — до вызова дорогой модели.
4. Загрузка **профиля** (реляция + JSONB) и контекста сценария; при необходимости — **RAG** из векторного хранилища.
5. Запуск **графа агента**; роутер выбирает модель (текст vs изображение).
6. Сохранение сообщений / артефактов; инкремент **usage** при успешной генерации.
7. Формирование **OutboundMessage**; адаптер отправляет в канал.

## Поток: оплата (Stars)

1. Пользователь инициирует оплату в Telegram → адаптер получает `pre_checkout_query` / успешный платёж.
2. Вызов общего модуля **Billing**: валидация, идемпотентная запись транзакции, обновление плана пользователя в **Postgres**.
3. Уведомление пользователя через **OutboundMessage**.

## Расширения без ломки ядра

- Новый агент / вертикаль: конфиг + запись в `agent_verticals` (или аналог), отдельные правила валидации JSONB при необходимости.
- Новый канал: новый адаптер + регистрация в `channel_links`.
- Новый платёжный провайдер: новая реализация `BillingProvider`.
- Новый тип лимита: новая строка в `plan_limits` и ресурс в коде учёта usage.
